# NCDNTS Incident Research

Research date: 2026-02-27
Source: JIRA project NCDNTS (Incidents - Tier 2 Escalation)
Period analyzed: 2025-08-27 to 2026-02-27 (6 months)

---

## 1. Overview

The NCDNTS project is a Jira Service Management project for production incidents. Tickets are auto-generated by Observe monitors, Datadog monitors, CI/CD workflow monitors, and bot-driven error detection systems. The ticket numbering reached NCDNTS-12856 by end of the research period, with tickets in the 10000-12856 range covering the 6-month window.

**Key statistics:**
- Estimated total incidents in 6 months: ~2,800+ tickets (based on range NCDNTS-10000 to NCDNTS-12856)
- The vast majority are automated alerts, many of which are transient and auto-close
- Priority distribution observed: predominantly Medium, with notable High-priority clusters around telemetry outages and customer-facing errors
- No Highest-priority tickets were found in the 6-month window
- Common statuses: New, Acknowledged, In Progress, Mitigated, Closed

**Incident sources (how tickets are created):**
1. **Observe monitors** -- telemetry outages, engine crashes, OOM brake, transaction aborts, Snowflake Platform Metrics
2. **Observe ERP monitors** -- SPCS control plane errors (automated with error codes)
3. **CI/CD workflow monitors** -- deployment failures, test ring failures
4. **Bot-driven error detection** -- "Error in monitored account" tickets auto-created when known bugs hit production accounts
5. **Datadog monitors** -- infrastructure alerts (CosmosDB capacity, etc.)
6. **Wiz security scanner** -- vulnerability and anomaly detection
7. **Manual filing** -- customer-reported issues and oncall-initiated tickets

## 2. Incident Type Taxonomy

### 2.1 Engine Crash (SPCS)

**Description:** An engine running in Snowflake SPCS crashes (segfault, stack overflow, abort, or other fatal error). The engine container restarts but loses all state.

**Typical symptoms:**
- Observe alert: "SPCS: The engine X crashed in the account Y"
- Error logs matching "segmentation fault" and "signal"
- Engine container restart detected via `spcs.container.restarts.total` metric

**Frequency:** Very high -- dozens per week across all accounts. The most common incident type by volume.

**Typical severity:** Medium priority

**Typical environments:** spcs-prod, spcs-int

**Owning team:** Julia team (segfaults), Storage team (storage/network stack), Backend team (stack overflows)

**Resolution pattern:**
1. Check if crash was due to segfault (look for "segmentation fault" in error logs)
2. If segfault, retrieve core dump using instructions at the core dump analysis runbook
3. Identify the crash component from logs and route to appropriate team
4. Most are transient; engine auto-restarts and resumes

**Referenced runbooks:**
- [Core dump analysis](https://relationalai.atlassian.net/wiki/x/BYD2Fg) -- retrieving and reading segfault core dumps

**Referenced Observe dashboards/monitors:**
- [Alert details](https://171608476159.observeinc.com/workspace/41759331/alert) -- per-alert details
- [Threshold monitor 42938640](https://171608476159.observeinc.com/workspace/41759331/threshold-monitor/42938640) -- engine crash detection

**Example tickets:** NCDNTS-12856, NCDNTS-12855, NCDNTS-12854, NCDNTS-12853

### 2.2 Transaction Aborts with Reason "Engine Failed"

**Description:** Transactions are aborted by the ERP because it did not receive a heartbeat from the engine for 20 minutes. Root causes include engine crash, OOM, brownout, Snowflake maintenance, or long heartbeat requests.

**Typical symptoms:**
- Observe alert: "Transactions were aborted with reason 'engine failed' on the engine X"
- Multiple transactions may be affected simultaneously
- Engine may or may not restart

**Frequency:** High -- multiple per week

**Typical severity:** Medium priority

**Owning team:** ERP team (initial triage), then routed to Julia/Storage/Backend teams based on root cause

**Resolution pattern:**
1. Open the [Engine failures dashboard](https://171608476159.observeinc.com/workspace/41759331/dashboard/Engine-failures-41949642) with the transaction ID, engine name, and account ID
2. Follow the 3-step investigation process (filter params, zoom to timeframe, inspect cards)
3. Check the "Engine last termination reason" card for crash/OOM indicators
4. Check "Server heartbeats per second" for brownout indicators
5. Check "Service lifecycle events" for user-initiated operations
6. Check for Snowflake scheduled maintenance (Mon-Thu 11PM-5AM local time)

**Referenced runbooks:**
- [How to investigate transaction aborts with reason "engine failed" in SPCS](https://relationalai.atlassian.net/wiki/spaces/ES/pages/806748161) -- comprehensive guide with step-by-step dashboard usage

**Referenced Observe dashboards/monitors:**
- [Engine failures dashboard (41949642)](https://171608476159.observeinc.com/workspace/41759331/dashboard/Engine-failures-41949642) -- primary investigation tool
- [Count monitor 42297913](https://171608476159.observeinc.com/workspace/41759331/count-monitor/42297913) -- transaction abort detection
- Engine failures DataDog dashboard (backup): `5u7-367-vkv`

**Root cause sub-categories:**
- **Engine crash** (segfault, stack overflow, abort) -- check "Engine last termination reason" card for `Failed` or `Done`
- **OOM** -- check for `FailedWithOOM` label; inspect Jemalloc profiles and CPU profiles
- **Brownout** -- server heartbeats drop below 1/sec; check Julia GC time and Julia compilation time
- **Long heartbeat requests** -- heartbeat task takes >20 min; known unknown, engage ERP team
- **Engine lifecycle events** -- user deleted/suspended/upgraded engine during transaction
- **Snowflake maintenance** -- scheduled downtime Mon-Thu 11PM-5AM local time
- **PyRel test brownouts** -- specific to XS engines in spcs-int (NCDNTS-4522)

**Example tickets:** NCDNTS-12840, NCDNTS-12839, NCDNTS-12834

### 2.3 OOM Brake Transaction Cancellations

**Description:** The OOM brake mechanism proactively cancels transactions when memory pressure is detected, before the OS would kill the engine process.

**Typical symptoms:**
- Observe alert: "Repeated transaction cancellations by the OOM brake on the engine X"
- Multiple transactions cancelled in sequence
- Engine remains running (unlike a full OOM kill)

**Frequency:** Moderate -- several per month across different accounts

**Typical severity:** Medium priority

**Recurring accounts:** `by_dev_ov40102` (most frequent), `rai_studio_sac08949`, `att_cdononprod_cdononprod`, `rai_mirror_x_aqb71127`, `ey_fabric233_rua08657`

**Resolution pattern:**
1. Identify cancelled transactions and concurrent transactions
2. Note the stage at which transactions were cancelled
3. Note the queries being evaluated
4. Inspect metrics using the [OOM Investigations dashboard](https://171608476159.observeinc.com/workspace/41759331/dashboard/OOM-Investigations-41777956)
5. Check Jemalloc absolute and relative profiles near OOM events
6. Check CPU profiles to identify hot code paths

**Referenced Observe dashboards/monitors:**
- [OOM Investigations dashboard (41777956)](https://171608476159.observeinc.com/workspace/41759331/dashboard/OOM-Investigations-41777956)
- [Threshold monitor 42445898](https://171608476159.observeinc.com/workspace/41759331/threshold-monitor/42445898) -- OOM brake detection

**Example tickets:** NCDNTS-12357, NCDNTS-12352, NCDNTS-12317, NCDNTS-12201, NCDNTS-11826

### 2.4 Telemetry Outage (Observe)

**Description:** No telemetry data received in Observe from one or more regions. Can be caused by Snowflake event table issues, O4S task problems, or Observe platform issues.

**Typical symptoms:**
- Observe alert: "[Observe] Telemetry outage in REGION_NAME"
- No data flowing into Observe dashboards for the affected region
- Multiple monitors across the region may trigger simultaneously

**Frequency:** Moderate -- clusters of multiple regions affected at once; roughly weekly occurrences

**Typical severity:** High priority

**Affected regions observed:** AWS_US_WEST_2, AWS_US_EAST_1, AWS_EU_WEST_1, AWS_EU_CENTRAL_1, AWS_AP_SOUTHEAST_2, AZURE_WESTUS2, AZURE_WESTEUROPE

**Resolution pattern (from the Runbooks page):**
1. Check [Observe Status Page](https://status.observeinc.com/) for platform outage
2. Verify telemetry in event table of O4S account:
   ```sql
   SELECT * FROM TELEMETRY.TELEMETRY.SHARED_EVENTS
   WHERE TIMESTAMP > DATEADD(minute, -20, CURRENT_TIMESTAMP())
   ORDER BY TIMESTAMP DESC LIMIT 100;
   ```
3. If no telemetry: file Sev-1 support case with Snowflake
4. If telemetry exists: check O4S ingestion task status
5. If task stuck: cancel it with `SYSTEM$CANCEL_QUERY`
6. If no running tasks: check if O4S app is installed; if yes, raise incident with Observe
7. Notify `@reliability` in `#helpdesk-observability`

**Referenced runbooks:**
- [Runbooks (SPCS space)](https://relationalai.atlassian.net/wiki/spaces/SPCS/pages/1697054722/Runbooks) -- comprehensive runbook for all telemetry outage types
- [Snowflake support case instructions](https://relationalai.atlassian.net/wiki/spaces/SPCS/pages/546668801)
- [ACCOUNTADMIN access workflow](https://relationalai.atlassian.net/wiki/spaces/SPCS/pages/1441857538)

**Referenced Observe dashboards/monitors:**

### 2.5 Snowflake Platform Metrics Outage

**Description:** Snowflake Platform Metrics specifically stop flowing to Observe, while other telemetry may still work.

**Typical symptoms:**
- Observe alert: "[Observe] Snowflake Platform Metrics outage in REGION_NAME"
- Platform metrics missing for 4+ hours

**Frequency:** Low -- approximately 3 in 6 months

**Typical severity:** High priority

**Resolution pattern:**
1. Check Observe status page
2. Log into O4S account and verify platform metrics:
   ```sql
   SELECT * FROM TELEMETRY.TELEMETRY.SHARED_EVENTS
   WHERE RECORD_TYPE = 'METRIC'
   AND SCOPE['name'] = 'snow.spcs.platform'
   AND TIMESTAMP > DATEADD('hour', -4, current_timestamp())
   LIMIT 100;
   ```
3. If missing: file Sev-1 Snowflake support case
4. Notify `@reliability` in `#helpdesk-observability`

**Referenced runbooks:**
- [Runbooks - Snowflake Platform Metrics Outage section](https://relationalai.atlassian.net/wiki/spaces/SPCS/pages/1697054722/Runbooks#Runbook-for-Snowflake-Platform-Metrics-Outage-monitor)

### 2.6 ERP Errors (SPCS Control Plane)

**Description:** The Engine Resource Provider (ERP), aka the SPCS control plane, encounters errors. These are automated incidents with structured error codes.

**Typical symptoms:**
- Alert: "[ERP]: erp_{component}_{upstream}_{reason} error happens in account :X"
- Structured error code, error message, failing component, and upstream dependency provided
- Links to Observe traces and logs included automatically

**Frequency:** Very high -- multiple per day. Most common ERP error types:
- `erp_txnevent_internal_stream_write_error` -- S3/blob throttling (usually transient, no customer impact)
- `erp_blobgc_sf_unknown` -- blob GC Snowflake query errors (usually transient)
- `erp_spcs_internal_request_reading_error` -- request reading failures
- `erp_enginerp_sf_oauth_token_expired` -- OAuth token expiry
- `erp_logicrp_sf_invalid_image_in_spec` / `erp_internallogicrp_sf_invalid_image_in_spec` -- post-upgrade image unavailable
- `erp_modelerrp_sf_unknown` -- model error RP Snowflake errors
- `erp_spcs_sf_metadata_health_check_error` -- metadata health check failures
- `erp_blobgc_sf_oauth_token_expired` -- blob GC OAuth expiry
- `erp_engine_enginepending` -- engine not ready when transaction arrives
- `erp_enginerp_internal_engine_provision_timeout` -- engine provisioning timeout
- `erp_txnrp_internal_db_init_failed` -- database init race condition

**Typical severity:** Medium priority

**Resolution pattern:**
1. Read error code from the alert (format: `erp_{component}_{upstream}_{reason}`)
2. If "unknown" in error code: report to ERP team in `#team-prod-engine-resource-providers-spcs`
3. Open [ERP error runbook](https://relationalai.atlassian.net/wiki/spaces/ES/pages/658407425/Actionable+ERP+monitoring+Run+book) and search for the error code
4. Check trace details in Observe for the specific trace_id
5. Check logs for affected transaction/query
6. Determine if transient (encounter count < 2) or persistent
7. If persistent, escalate to ERP team

**Referenced runbooks:**
- [Actionable ERP monitoring Run book](https://relationalai.atlassian.net/wiki/spaces/ES/pages/658407425/Actionable+ERP+monitoring+Run+book) -- comprehensive error code reference with diagnostic steps for each error type
- [ERP monitoring overview](https://relationalai.atlassian.net/wiki/spaces/ES/pages/655491086/Actionable+ERP+monitoring+on+Observe)
- [ERP diagnostic 101](https://relationalai.atlassian.net/wiki/spaces/ES/pages/756023297)

**Referenced Observe dashboards/monitors:**
- [ERP actionable monitor v2 (42488209)](https://171608476159.observeinc.com/workspace/41759331/promote-monitor/42488209) -- primary ERP error detection
- [Trace explorer](https://171608476159.observeinc.com/workspace/41759331/trace-explorer?spanDatasetId=41867217) -- trace investigation
- [Log explorer](https://171608476159.observeinc.com/workspace/41759331/log-explorer?datasetId=41832558) -- log investigation

**Escalation Slack channel:** `#team-prod-engine-resource-providers-spcs`

**Example tickets:** NCDNTS-12821, NCDNTS-12815, NCDNTS-12793, NCDNTS-12777, NCDNTS-12775

### 2.7 CI/CD Workflow Failures

**Description:** Deployment or test workflows fail in GitHub Actions. Two main sub-types: Snowflake Native App deployment failures and rai-ci test ring failures.

**Sub-type A: Snowflake Native App Deployment Failures**

**Typical symptoms:**
- Alert: "[NA Deployments] workflow `Snowflake Native App & SPCS Continuous Deployment to spcs-int (US WEST region)` failed"
- Or: "[PyRel v0] workflow ... failed"
- Or: "[NA Integration] workflow ... failed"
- Or: "[Engine Resource Providers] workflow ... failed"
- GitHub Actions job links provided

**Frequency:** Very high -- multiple per day (often transient)

**Resolution pattern:**
1. Follow the [Deployment failure incidents runbook](https://relationalai.atlassian.net/wiki/x/AQBrWQ)
2. Check the specific failed GitHub Actions job
3. Retry or investigate the root cause

**Sub-type B: rai-ci Test Ring Failures**

**Typical symptoms:**
- Alert: "[EngineOperations] workflow `rai-ci - Test Ring 3` failed"

**Frequency:** High -- multiple per week

**Resolution pattern:** Check the CI workflow for test failures, flaky tests, or infrastructure issues

**Referenced runbooks:**
- [Deployment failure incidents runbook](https://relationalai.atlassian.net/wiki/x/AQBrWQ)

**Example tickets:** NCDNTS-12827, NCDNTS-12822, NCDNTS-12818, NCDNTS-12816, NCDNTS-12814

### 2.8 Poison Commits

**Description:** A commit that breaks the build/test pipeline is detected and flagged. Requires immediate revert or forward-fix.

**Typical symptoms:**
- Alert: "Poison commit HASH is added to repository REPO for cloud CLOUD"
- Workflow failure link provided
- Impact assessment included (prod affected? staging affected?)

**Frequency:** Moderate -- roughly 2-3 per month

**Typical severity:** Medium priority

**Affected repositories observed:** `spcs-control-plane`, `raicode`, `spcs-sql-lib`

**Resolution pattern (from ticket template):**
1. Ensure poison commit is reverted or forward-fixed (prefer revert)
2. If already released: ensure rollback or hotfix
3. Link test and observability repairs to the incident
4. Answer post-mortem questions:
   - How was the poison commit identified?
   - Why was it a poison commit?
   - Why wasn't this caught earlier?
   - What will we do differently next time?
   - Follow-ups needed?

**Example tickets:** NCDNTS-12300, NCDNTS-12161, NCDNTS-11958, NCDNTS-11858, NCDNTS-11857

### 2.9 Errors in Monitored Accounts (Bot-Generated)

**Description:** A known or new error occurs in a monitored (typically customer) account and is escalated to an incident by a bot. The bot links to the originating JIRA bug ticket (RAI-XXXXX).

**Typical symptoms:**
- Alert: "Error in monitored account due to RAI-XXXXX: [error description]"
- Or: "New error in monitored account (fingerprint: 0xHEXHASH)"
- Includes affected accounts, engine versions, databases, environments
- Example transaction ID and Observe log link provided

**Frequency:** High -- several per week

**Error types observed:**
- `SafeErrorException: Database was closed and this transaction was discarded` (RAI-47787)
- `Bug/race condition in handling aborted/completed state` (RAI-46881)
- `Final VO does not match with its free variables` (RAI-35892)
- `CompositeException (2 tasks)` (RAI-47650)
- `BoundsError: attempt to access N-codeunit String at index [M]` (RAI-43400)
- `AssertionError: output_arity called on HigherOrderPrimitiveInfo!` (RAI-47645)
- Flaky test detections in raicode (RAI-40786)

**Resolution pattern:**
1. Check if the RAI ticket already has a fix in progress
2. Review affected accounts and engine versions
3. Check Observe logs for the example transaction
4. If new error: triage and assign to appropriate team
5. If known error with fix deployed: verify fix and close

**Example tickets:** NCDNTS-12826, NCDNTS-12792, NCDNTS-12683, NCDNTS-12557, NCDNTS-12536

## 9. Resolution Patterns

**Common mitigation strategies observed:**

1. **Auto-recovery / Transient** -- The most common resolution. Many ERP errors, engine crashes, and telemetry blips self-resolve. Oncall verifies it was transient (encounter count < 2) and closes.

2. **Engine restart** -- For crashes and OOMs, the engine container auto-restarts. The oncall monitors for recurrence.

3. **Task cancellation** -- For stuck O4S ingestion tasks, cancel the stuck query and a new task is auto-triggered.

4. **Snowflake support case** -- For telemetry outages confirmed to be Snowflake-side, file a Sev-1 case.

5. **Observe support case** -- For O4S app issues, raise incident with Observe.

6. **Code revert** -- For poison commits, revert the offending commit.

7. **Escalation to owning team** -- Route to ERP team, Julia team, Storage team, or Backend team based on root cause.

8. **Warehouse upsizing** -- For Observe ingestion lag, upsize the `observability_wh`.

9. **Duplicate/known issue closure** -- For known bugs (e.g., post-upgrade image spec errors), link to the tracking ticket and close.

## 13. Gaps and Recommendations

### Gaps Identified

1. **No runbook for engine crashes** -- Engine crash tickets reference the core dump analysis page but lack a comprehensive triage runbook. The oncall must rely on ad-hoc investigation.

2. **No runbook for OOM brake incidents** -- OOM brake tickets point to the OOM Investigations dashboard but lack documented resolution steps beyond "inspect metrics."

3. **No runbook for poison commits** -- Poison commit tickets contain an inline procedure but there is no centralized runbook.

4. **No runbook for "Error in monitored account" incidents** -- These bot-generated tickets require the oncall to understand the originating RAI ticket and determine impact, with no standard procedure.

5. **ERP runbook incomplete** -- The ERP runbook acknowledges "we unfortunately haven't covered all error codes yet." Several error codes lack documented mitigation steps.

6. **No automated severity assessment** -- All engine crashes and transaction aborts default to Medium priority regardless of whether they affect production customers or internal test accounts.

7. **High noise-to-signal ratio** -- Many incidents are transient and auto-resolve. The oncall must manually check encounter counts to distinguish transient from persistent issues.

8. **No customer impact classification** -- Incidents do not clearly indicate whether production customers are affected vs. internal test accounts.

9. **Missing region information in many tickets** -- Engine crash and OOM brake tickets do not consistently include the deployment region.

10. **Duplicate ticket flooding** -- The same engine crash or ERP error can generate many near-identical tickets within minutes (e.g., 4+ engine crash tickets for the same engine within 2 minutes).

### Recommendations for the Oncall Skill

1. **Automated triage routing** -- Use error code patterns, account names, and environment to auto-suggest the owning team and relevant runbook.

2. **Transient detection** -- Auto-check encounter counts in Observe to flag incidents likely to be transient.

3. **Customer impact assessment** -- Maintain a list of monitored/production accounts to distinguish customer-facing from internal incidents.

4. **Runbook auto-linking** -- Map incident summary patterns to specific runbook sections.

5. **Duplicate detection** -- Identify when multiple tickets describe the same underlying event.

6. **Investigation quickstart** -- Pre-populate Observe dashboard URLs with the correct parameters from the ticket data.
